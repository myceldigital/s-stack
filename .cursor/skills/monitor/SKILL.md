# Monitor

## Mission

Tell the operator what actually landed, what drifted, and what needs follow-up.

## Use this skill when

- the user explicitly asks for `/monitor` or the equivalent workflow
- the operating context matches this skill's mission more closely than a generic drafting pass
- the result needs to be stored or reasoned about as part of the Comms-Stack system

## Read before acting

- the launched campaign or incident materials
- `knowledge/stripe-narrative.md`
- relevant relationship or outlet records

## Workflow

1. Summarize the outputs that actually went live.
2. Assess how the message was framed externally.
3. Identify corrections, clarifications, or follow-up opportunities.
4. Compare outcomes to the intended thesis.
5. Surface reusable lessons, not just counts.

## Output contract

Use the relevant repo template when it helps. The output should include:

- coverage summary
- message pull-through
- misframing list
- correction opportunities
- follow-up actions
- reusable lessons

## Quality bar

- Do not confuse activity with narrative movement.
- A single important misframe can matter more than many pickups.
- Name what changed because of the work.

## Escalate immediately when

- meaningful factual errors in coverage
- narrative drift that affects trust
- unexpected backlash or competitor counter-framing

## Memory update rule

Feed durable lessons into campaign archive, media graph, or narrative memory as appropriate.

## Do not

- Do not produce vanity reporting.
- Do not count weak placements as proof of strategic success.
