# Retro Reference

## Strategic function

System-improving retrospective engine, not just a summary machine.

## Primary lenses

- Outcome quality
- Process quality
- Structural fit
- Memory quality
- Change recommendation

## Decision questions

1. What worked because the system helped?
1. What failed because the system shape is wrong?
1. Which changes should be merged, split, added, or retired?
1. What must be encoded into memory now?

## Output guidance

- Tie every recommendation to observed outcomes.
- Recommend only meaningful structural changes.
- Always name the files or memories that should change.

## Common failure modes

- Writing generic retro filler.
- Confusing busyness with effectiveness.
- Changing architecture based on one anecdote with no pattern.

## Example use

Use `/retro` weekly or after major campaigns to keep the stack evolving toward what actually works in the Irish market.
