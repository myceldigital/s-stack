# Retro

## Mission

Convert lived operating experience into better system design.

## Use this skill when

- the user explicitly asks for `/retro` or the equivalent workflow
- the operating context matches this skill's mission more closely than a generic drafting pass
- the result needs to be stored or reasoned about as part of the Comms-Stack system

## Read before acting

- latest campaign, incident, and monitoring outputs
- `knowledge/weekly-retros/`
- relevant archive entries

## Workflow

1. Identify what worked and what failed with evidence.
2. Name surprises and hidden constraints.
3. Decide whether the issue was execution, structure, timing, or memory quality.
4. Recommend merges, splits, additions, or retirements only when justified.
5. State what knowledge files must change.

## Output contract

Use the relevant repo template when it helps. The output should include:

- what worked
- what failed
- what surprised us
- structural recommendations
- knowledge updates required
- next operating changes

## Quality bar

- Tie recommendations to outcomes, not vibes.
- Prefer a small number of structural changes with clear rationale.
- A retro that teaches nothing is a failed retro.

## Escalate immediately when

- repeat failure mode across weeks
- agent contract no longer fits reality
- memory layer has become the bottleneck

## Memory update rule

This skill is one of the main compounders of the system. Be explicit about updates.

## Do not

- Do not write bland retrospective filler.
- Do not confuse effort with effectiveness.
